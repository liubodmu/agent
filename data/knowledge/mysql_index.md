# MySQL 索引面试知识点

## 索引是什么

索引是帮助 MySQL 高效获取数据的数据结构。

类比：书的目录，通过目录快速定位到内容页码。

## 索引的数据结构

### B+ 树（InnoDB 默认）
- 所有数据存储在叶子节点
- 叶子节点用双向链表连接
- 非叶子节点只存索引，一个节点能存更多 key
- 树高度一般 2-4 层，每次查询 IO 次数少

### 为什么用 B+ 树而不是 B 树？
1. B+ 树非叶子节点不存数据，同样大小的节点能存更多 key → 树更矮 → IO 更少
2. 叶子节点链表连接 → 范围查询高效
3. 所有查询都要到叶子节点 → 查询时间稳定

### 为什么不用红黑树？
- 红黑树是二叉树，数据量大时树太高，IO 次数太多
- B+ 树是多叉树，树高可控

## 索引类型

### 聚簇索引（主键索引）
- InnoDB 的主键索引，叶子节点存储整行数据
- 一张表只能有一个聚簇索引

### 非聚簇索引（二级索引）
- 叶子节点存储主键值
- 查询时需要"回表"：先查二级索引得到主键，再查主键索引得到数据

### 覆盖索引
- 查询的列全部在索引中，不需要回表
- 性能最好，explain 中 Extra 显示 "Using index"

### 联合索引
- 多个列组合的索引
- 遵循最左前缀原则

## 最左前缀原则

联合索引 (a, b, c)：
- WHERE a=1 ✓ 走索引
- WHERE a=1 AND b=2 ✓ 走索引
- WHERE a=1 AND b=2 AND c=3 ✓ 走索引
- WHERE b=2 ✗ 不走索引
- WHERE a=1 AND c=3 ✓ 只用到 a 列索引

## 索引失效场景

1. 对索引列使用函数：WHERE YEAR(create_time) = 2024
2. 隐式类型转换：varchar 列用 int 查询
3. LIKE 以 % 开头：WHERE name LIKE '%张'
4. OR 条件中有非索引列
5. NOT IN、NOT EXISTS（部分场景）
6. 索引列参与计算：WHERE age + 1 = 18

## EXPLAIN 关键字段

- type：查询类型，性能 system > const > eq_ref > ref > range > index > ALL
- key：实际使用的索引
- rows：预估扫描行数
- Extra：额外信息（Using index、Using filesort、Using temporary）

## 面试常见追问

Q: 什么时候不该建索引？
A: 1.数据量小 2.频繁更新的列 3.区分度低的列（如性别）

Q: 索引越多越好吗？
A: 不是。索引占空间，增删改时需要维护索引，影响写入性能。

Q: 如何优化慢查询？
A: 1.EXPLAIN 分析 2.添加合适索引 3.避免 SELECT * 4.分页优化 5.减少子查询
